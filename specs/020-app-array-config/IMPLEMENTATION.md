# Implementation Summary: Pure Data Application Configuration

**Feature**: 020-app-array-config\
**Branch**: `020-app-array-config`\
**Status**: ✅ Implemented\
**Approach**: Pure Data Pattern (Option A - refined)

## Overview

User configurations are now pure data attribute sets with no imports or helper functions. Platform libraries automatically load user data, extract the applications array, and generate the necessary imports.

## Final Implementation

### User Configuration Pattern

**File**: `user/<username>/default.nix`

```nix
{ ... }:

{
  user = {
    name = "username";
    email = "user@example.com";
    fullName = "Full Name";
    applications = [ "git" "zsh" "helix" ];
  };
}
```

**Key characteristics**:

- No imports
- No let bindings
- No helper functions
- Pure data only
- Simple attribute set

### Platform Library Implementation

**File**: `platform/darwin/lib/darwin.nix` (macOS example)

```nix
# Import discovery for application module generation
discovery = import ../../shared/lib/discovery.nix { inherit lib; };

mkDarwinConfig = { user, profile, system ? "aarch64-darwin" }: let
  # Load user configuration as pure data (before module evaluation)
  userDataPath = ../../../user/${user};
  userData = import userDataPath { };
  
  # Extract applications from user data (simple attribute access)
  userApplications = userData.user.applications or [];
  
  # Generate applications module at platform lib level
  appsModule = if userApplications != [] then
    discovery.mkApplicationsModule {
      inherit lib;
      applications = userApplications;
    }
  else
    { };
in
  inputs.nix-darwin.lib.darwinSystem {
    # ... system config ...
    modules = [
      # ... other modules ...
      {
        home-manager.users.${user} = {
          imports = [
            userData      # Pure user data
            appsModule    # Generated by platform lib
            ../../../user/shared/lib/home-manager.nix  # Bootstrap
            # ... other imports ...
          ];
        };
      }
    ];
  };
```

**Key implementation details**:

1. User data loaded as plain import (not through module system)
1. Applications extracted via attribute access before module evaluation
1. mkApplicationsModule called at platform lib level
1. Both userData and appsModule combined in imports

### Home Manager Bootstrap

**File**: `user/shared/lib/home-manager.nix`

Provides the `user.*` options including `user.applications`. No longer exports helper functions - platform libs handle everything.

```nix
{ config, lib, pkgs, ... }:

{
  options.user = {
    name = lib.mkOption { type = lib.types.str; ... };
    email = lib.mkOption { type = lib.types.str; ... };
    fullName = lib.mkOption { type = lib.types.str; ... };
    
    applications = lib.mkOption {
      type = lib.types.nullOr (lib.types.listOf lib.types.str);
      default = null;
      description = ''
        List of application names to import from the platform application registry.
        Platform libraries automatically read this field and generate imports.
      '';
    };
    
    # ... other options ...
  };
  
  config = { /* ... */ };
}
```

## Technical Achievements

### ✅ Solved Infinite Recursion

**Problem**: Cannot reference `config` in `imports` (Nix limitation)

**Solution**: Extract data BEFORE module evaluation

- Load user file as plain import
- Access `.user.applications` as attribute (not config)
- Generate imports before module system sees them

### ✅ True Automation

Users don't call any functions or manage any imports. Platform libs handle everything automatically.

### ✅ Backward Compatible

Old pattern with explicit discovery imports still works. This is purely additive.

### ✅ DRY Principle

Applications declared exactly once in `user.applications`. No duplication anywhere.

## Architecture

### Data Flow

```
1. User writes pure data config
   ↓
2. Platform lib imports file (plain import, not module eval)
   ↓
3. Platform lib extracts .user.applications (attribute access)
   ↓
4. Platform lib calls discovery.mkApplicationsModule
   ↓
5. Platform lib combines userData + appsModule in home-manager imports
   ↓
6. Module system evaluates combined configuration
   ↓
7. Applications are configured
```

### Key Insight

The critical insight that makes this work:

```nix
# This is NOT module evaluation - it's just importing a .nix file
userData = import ../../../user/${user} { };

# This is NOT config.user.applications - it's plain attribute access
apps = userData.user.applications;

# No circular dependency because we're not in the module system yet
appsModule = mkApps apps;

# NOW we enter the module system with both pieces
imports = [ userData appsModule ];
```

## Design Decisions

### Why Pure Data Over mkApps Helper?

| Aspect | mkApps Helper | Pure Data | Winner |
|--------|---------------|-----------|--------|
| User code | 9 lines | 7 lines | Pure Data |
| Nix knowledge | Moderate | Minimal | Pure Data |
| Imports | Yes (mkApps) | No | Pure Data |
| Truly automatic | No (explicit call) | Yes | Pure Data |
| Platform complexity | Low | Medium | mkApps |
| Achieves original vision | Partial | Complete | Pure Data |

**Decision**: Pure Data

- Better UX (simpler for users)
- Achieves original feature goal ("automatic")
- Only slight increase in platform lib complexity
- Still maintainable and well-documented

### Why No Backward Compatibility Layer?

**Decision**: Breaking change acceptable because:

1. Only 3 users in repository
1. Migration is trivial (remove imports, move apps to user struct)
1. Cleaner architecture without compatibility shims
1. Can document old pattern in case needed

## Lessons Learned

### Nix Module System Constraints

- Cannot reference `config` in `imports` (infinite recursion)
- Can import files and access attributes before module evaluation
- Module evaluation happens AFTER all imports are collected

### Pure Data Benefits

- Easier to reason about (no hidden logic)
- Simpler for users (less Nix knowledge needed)
- Better separation of concerns (data vs logic)
- More testable (can validate data independently)

### Platform Lib Patterns

- Can pre-process user data before module evaluation
- Good place for orchestration logic
- Users benefit from not seeing complexity

## Future Enhancements

### Potential Improvements

1. **Type validation at platform lib level**: Validate `user.applications` is a list of strings before passing to discovery
1. **Better error messages**: Catch common mistakes early with clear messages
1. **NixOS platform support**: Apply same pattern to nixos.nix
1. **Home Manager standalone**: Extend to standalone home-manager configs
1. **Application groups**: Allow `applications = [ "@dev" "@shell" ]` for predefined groups

### Not Recommended

1. **Automatic import without declaration**: Would make configs harder to understand
1. **Configuration in flake.nix**: Violates user/system separation
1. **Dynamic application loading**: Too much magic, harder to debug

## Conclusion

The pure data approach successfully achieves the original feature vision while providing superior user experience compared to the mkApps helper pattern. Users write only configuration data, platform libraries handle all the complexity, and the system gracefully avoids the infinite recursion constraint through pre-evaluation data extraction.
