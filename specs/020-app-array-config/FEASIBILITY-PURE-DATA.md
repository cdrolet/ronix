# Feasibility Analysis: Pure Data User Configuration

**Date**: 2025-12-02\
**Feature**: 020-app-array-config Alternative Approach\
**Analyst**: Claude (via user request)\
**Status**: ✅ FEASIBLE

## Executive Summary

**Question**: Can we simplify user configuration further by making user files pure data (no imports) and having flake.nix/platform libs handle all imports?

**Answer**: **YES - This is technically feasible and avoids the infinite recursion problem.**

The proposed approach works because:

1. User files can be pure attribute sets (just data, no imports)
1. Platform libs can `import` user files and extract `.user.applications` BEFORE module evaluation
1. Platform libs can generate the mkApps module based on extracted data
1. Both user data and generated apps module are combined in `home-manager.users.${user}.imports`
1. No circular dependency because data extraction happens BEFORE config evaluation

## Proposed Architecture

### User Configuration (Pure Data)

**Current** (020 implementation with mkApps helper):

```nix
{ config, pkgs, lib, userContext, ... }:

let
  mkApps = import ../shared/lib/mk-apps.nix { inherit lib; };
  applications = [ "git" "zsh" "helix" ];
in
{
  imports = [
    ../shared/lib/home-manager.nix
    (mkApps applications)
  ];
  
  user = {
    name = "cdrolet";
    email = "cdrolet@work.example.com";
    fullName = "Charles Drolet";
    inherit applications;
  };
}
```

**Proposed** (Pure data - no imports, no let, no mkApps):

```nix
{ ... }:

{
  user = {
    name = "cdrolet";
    email = "cdrolet@work.example.com";
    fullName = "Charles Drolet";
    applications = [ "git" "zsh" "helix" ];
  };
}
```

### Platform Library Changes

**Current** (`platform/darwin/lib/darwin.nix`):

```nix
home-manager.users.${user} = {
  imports = [
    ../../../user/${user}  # User config handles its own imports
  ];
};
```

**Proposed** (`platform/darwin/lib/darwin.nix`):

```nix
# Load user data BEFORE module evaluation
let
  userData = import ../../../user/${user} { };
  userApps = userData.user.applications or [];
  
  # Generate apps module at platform lib level
  discovery = import ../../shared/lib/discovery.nix { inherit lib; };
  appsModule = if userApps != [] then
    discovery.mkApplicationsModule {
      inherit lib;
      applications = userApps;
    }
  else
    { };
in
{
  home-manager.users.${user} = {
    imports = [
      # Pure user data (no imports)
      userData
      
      # Generated by platform lib
      appsModule
      
      # Home Manager bootstrap
      ../../../user/shared/lib/home-manager.nix
    ];
  };
}
```

### Why This Avoids Infinite Recursion

**Previous blocker** (automatic approach):

```nix
# FAILS: Cannot reference config in imports
imports = lib.optionals (config.user.applications != null) [
  (mkApps config.user.applications)  # ❌ Infinite recursion
];
```

**This approach** (pure data):

```nix
# WORKS: Extract data BEFORE module evaluation
let
  userData = import ./user.nix { };  # Just an import, not module eval
  apps = userData.user.applications;  # Just attribute access
in {
  imports = [
    userData       # Now becomes a module
    (mkApps apps)  # Generated from pre-extracted data
  ];
}
```

The key: `userData.user.applications` is accessed as a **plain attribute set**, not as `config.user.applications` through the module system.

## Benefits vs Current Approach

### Code Reduction Comparison

**Current (mkApps helper)**:

- 9 lines for user config (30% reduction from original)
- User needs to import mkApps
- User needs let binding
- User needs inherit applications

**Proposed (pure data)**:

- 7 lines for user config (56% reduction from original)
- User writes ONLY data
- Zero imports in user file
- Zero helper knowledge needed
- Truly "pure data"

### User Experience

**Current**:

```nix
let
  mkApps = import ../shared/lib/mk-apps.nix { inherit lib; };
  applications = [ "git" "zsh" ];
in {
  imports = [ ... (mkApps applications) ];
  user = { inherit applications; ... };
}
```

- User must know about mkApps
- User must understand let bindings
- User must understand imports
- User must use inherit to avoid repetition

**Proposed**:

```nix
{
  user = {
    name = "cdrolet";
    applications = [ "git" "zsh" ];
  };
}
```

- User just writes data
- No Nix knowledge required beyond attribute sets
- No helper functions to learn
- No imports to manage
- No duplication possible

## Implementation Complexity

### Changes Required

1. **Platform Libraries** (`platform/darwin/lib/darwin.nix`, etc.):

   - Add user data loading logic
   - Extract applications before module evaluation
   - Generate mkApps module
   - Adjust home-manager.users imports

1. **User Shared Libraries**:

   - Keep `user/shared/lib/home-manager.nix` (for options)
   - Remove `user/shared/lib/mk-apps.nix` (no longer needed by users)
   - Move mkApps logic to platform libs

1. **User Configurations**:

   - Simplify to pure data attribute sets
   - Remove all imports
   - Remove let bindings
   - Just user = { ... }

1. **Documentation**:

   - Update CLAUDE.md with new pattern
   - Update docs/features/020-app-array-config.md
   - Add migration guide

### Backward Compatibility

**Option A: Breaking change**

- All users must migrate to pure data
- Clean, consistent architecture
- One-time migration effort

**Option B: Support both patterns**

- Check if user file has imports
- If no imports: use new approach
- If imports: use old approach
- More complexity in platform libs

**Recommendation**: Option A (breaking change)

- Only 3 users to migrate
- Simpler implementation
- Better long-term maintainability
